Interpreting one result

How non-computational users can read a workflow output

A short guide for collaborators who need a clear, stepwise way to interpret a single run without technical details.

1. Workflow report Check whether the run completed, how many metadata fields were produced, and whether expert review is recommended.
2. Generated metadata Read the output by level: investigation, study, assay, sample, and observation unit (ISA-style structure).
3. Evaluation summary Use the evaluation summary to see required and overall completeness, and whether the output is schema-valid.
4. Expert interpretation Decide which suggested fields to accept, correct, or remove, with input from a domain expert where needed.

In plain language

The workflow identifies the type of research document.
It selects metadata templates that fit that document.
It fills them with values and flags stronger vs weaker evidence.
It checks whether the result is structurally complete enough for downstream use.

What to check first

Did the run complete?
Which fields are confirmed vs provisional?
Which metadata categories are present?
Which required fields are still missing?

Discussion

Metadata design for FAIR biology: questions for the community

We welcome input from the FAIR and biology community on these topics.

Minimum publishable metadata

For common study types in our domain, what is the minimum metadata set needed before a dataset becomes biologically reusable?
Which fields are strictly required for publication or repository submission, and which are desirable but optional?

Entity boundaries

How should we separate sample, host, pathogen, observation unit, and aggregate/community in complex studies?
Are there domain conventions in plant pathology and bioremediation we should adopt?

Controlled vocabularies

Which identifiers and vocabularies should be prioritized: taxonomy, environment, pollutant chemistry, host tissue, disease stage, treatment condition?
Which are already standard practice in the lab, even if not formally documented?

Source-of-truth materials

Which documents hold the most reliable metadata in practice: papers, lab notebooks, sample sheets, repository templates, or project proposals?
Which fields should never be inferred from narrative text alone?
These questions are intended to steer the discussion toward biology-driven metadata design. We aim to keep the focus on FAIR usability rather than implementation details.
Collaboration opportunities

Expertise and materials we seek from the community

We are looking for expert criteria and representative materials to align our workflow with community practice.

What we are looking for

1. A core checklist A lab- or community-validated list of required, recommended, and optional fields for the most relevant study types.
2. Example documents A small set of representative inputs: papers, sample sheets, proposals, metadata templates, or repository drafts.
3. Vocabulary priorities Guidance on which identifiers to prioritize: taxonomy, environmental terms, chemical names, disease stages, host tissues.
4. Expert review boundaries Which fields can be checked quickly by a biologist and which need deeper bioinformatics or data stewardship review.

A question for the community

“Could we define a domain-specific metadata checklist for common study types and test whether the current workflow captures the fields that matter for publication and reuse?”
For data stewards and biologists:

Which fields would you check first before trusting a generated metadata record from a study like ours?
We aim to leave with actionable, biology-led guidance. Specific, expert-driven input is more valuable than general feedback.
Field testing

Plans for validation in other institutions, with focus on bioremediation

We aim to validate usability and biological relevance in realistic metadata-generation settings across labs and institutions.

1. Select representative study types Soil or sediment remediation, wastewater or sludge remediation, and plant-microbe remediation systems.
2. Collect diverse inputs Papers, sample sheets, field logs, experimental protocols, repository templates, and project proposals.
3. Compare human vs system outputs Measure completeness, biological correctness, review burden, and readiness for sharing or submission.
4. Build a cross-lab consensus list Identify metadata fields every lab agrees are essential and note the ones that depend on local practice.

Bioremediation-specific metadata to prioritize

Pollutant identity and concentration
Matrix type: soil, sediment, sludge, water, rhizosphere
Treatment agent: microbial strain, consortium, plant species, amendment, carrier
Timepoints, spatial context, environmental conditions, and controls
Taxonomy and environmental terminology consistency

Most important validation question

Does the generated metadata help an external lab understand what was treated, where, when, under which conditions, and with what biological system?
Transition from internal evaluation to external use-case validation. Suited to collaboration and field-study planning.
Next steps

Proposed next steps and closing question

We propose to focus on clear, science-led next steps rather than implementation details.

Proposed next actions

Agree on one or two priority study types for domain-specific metadata checklists.
Select a small set of real documents for community or expert review.
Define which metadata fields must be manually confirmed before data sharing or publication.
Begin the first external bioremediation validation with a small, well-characterized dataset.

Closing message

We now have enough evidence to move from “can a workflow generate metadata?” toward “which biological metadata standards and field definitions are actually usable across labs?”
